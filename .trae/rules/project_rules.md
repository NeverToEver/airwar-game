# 编程架构与工程化规范（全局强制）

**本规范在所有代码生成、重构、设计、文档输出中强制执行，绝不违反。**
**对应 Skill：`architecture-enforcer`，在任何代码任务前自动激活。**

## 一、核心设计原则

1. **单一职责原则**：一个类/函数只做一件事，职责清晰可描述
2. **高内聚低耦合**：相关逻辑集中，无关逻辑隔离，模块间仅通过接口通信
3. **面向接口而非实现**：外部依赖抽象，不依赖具体细节，便于替换与扩展
4. **可维护性优先**：代码结构清晰、命名规范、注释适度、便于后续迭代
5. **可扩展性优先**：预留扩展点，避免硬编码、魔法数字、深层嵌套
6. **避免散弹式修改**：同类功能必须集中，不允许逻辑散落在多个文件

## 二、项目结构规范

1. 按功能/模块划分文件，不按"代码长短"划分
2. 入口文件只负责启动与调度，不包含业务逻辑
3. 核心模块独立成包/文件：窗口、渲染、输入、游戏逻辑、资源管理
4. 配置统一管理，不散落各处

## 三、代码风格强制规范

1. 命名清晰易懂，禁止无意义单字母变量（循环计数器 i/j 除外）
2. 函数长度不超过一屏（~40行），超出必须拆分
3. 避免深层嵌套，最多3层缩进，使用早返回/卫语句
4. 重复代码（3次以上）必须抽离为工具函数或基类
5. 全局变量最小化，状态集中管理

## 四、解耦强制要求（游戏/窗口项目尤其严格）

1. 窗口/渲染/事件必须封装为独立模块
2. 外部绝不直接操作底层库（如 pygame 窗口、surface 等）
3. 所有底层操作通过类接口暴露
4. 游戏逻辑、UI、输入、渲染相互隔离
5. 任何修改应只影响一个模块，不引发连锁改动

## 五、自动执行行为

1. 每次写代码前先做架构设计：模块划分、职责边界
2. 自动识别坏味道：散落逻辑、重复代码、混合职责
3. 自动提出重构建议并给出可运行代码
4. 自动生成模块设计文档、接口说明、维护指南
5. 若给出混乱代码，先指出问题，再按规范重构

## 六、输出格式要求

- 先说明设计思路与模块划分
- 再给出完整可运行代码
- 最后自动生成《模块设计文档》《接口说明》《维护指南》
- 始终保证代码可直接运行、结构干净、便于后期扩展
