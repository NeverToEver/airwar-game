# Air War 运行时资源优化技术说明

**日期**: 2026-04-17  
**状态**: 已完成  
**验证结果**: `python -m compileall airwar` 通过，`SDL_VIDEODRIVER=dummy pytest -q` 结果为 `161 passed, 1 warning`

---

## 一、背景与目标

当前项目运行时的资源消耗偏高，主要表现为以下两类问题：

1. 主循环内存在大量重复 `pygame.Surface(...)` 创建、渐变重绘和文本重复渲染。
2. 多个场景和 UI 模块各自维护相似绘制逻辑，缺少统一缓存策略，导致 CPU 与内存分配压力叠加。

本次优化目标不是做“单点修补”，而是建立一套可复用、可控制、可验证的运行时渲染优化方案，兼顾以下设计原则：

1. **单一职责**: 将 Surface 复用能力抽离到统一缓存模块，不让每个场景重复实现。
2. **开闭原则**: 新增缓存工具后，现有场景只需接入，不需要重写整套渲染逻辑。
3. **最小改动原则**: 优先优化热点路径，不随意改变游戏规则、输入逻辑和表现层职责边界。
4. **资源有界原则**: 缓存不是无限堆积，必须限制键粒度与最大容量，避免把 CPU 问题转成内存问题。

---

## 二、识别出的核心热点

### 2.1 重复创建 Surface 的热点

以下模块存在明显的逐帧 Surface 构造或重复绘制问题：

- `airwar/scenes/menu_scene.py`
- `airwar/scenes/login_scene.py`
- `airwar/scenes/pause_scene.py`
- `airwar/ui/reward_selector.py`
- `airwar/game/rendering/background_renderer.py`
- `airwar/game/rendering/game_renderer.py`
- `airwar/game/mother_ship/progress_bar_ui.py`
- `airwar/ui/buff_stats_panel.py`

### 2.2 手绘精灵热点

`airwar/utils/sprites.py` 中的玩家机、敌机、Boss、子弹均采用运行时手绘方式。  
这类函数虽然代码直观，但若每帧都重新画整套图形，会持续占用 CPU，并放大透明层和发光效果的开销。

### 2.3 高频但容易被忽略的热点

- Boss 警告文本在 `render()` 中重复创建字体渲染结果。
- Buff 面板每帧重新拼装底板、文字和汇总区。
- 进度条 UI 中母舰图标每帧重建。
- 渐变、粒子光晕、边框和面板发光在多个场景中重复出现，但此前没有统一复用层。

---

## 三、本次优化方案

### 3.1 引入统一 SurfaceCache

新增文件：

- `airwar/utils/render_cache.py`

该模块统一提供以下缓存能力：

- 垂直渐变缓存
- 矩形填充缓存
- 矩形边框缓存
- 矩形光晕缓存
- 粒子光晕缓存
- 圆形缓存
- 线条缓存
- 淡入淡出遮罩缓存

核心策略：

1. 通过 `_get_or_create()` 统一封装缓存命中与复用逻辑。
2. 对 `alpha` 做量化处理，避免细粒度透明度导致缓存键爆炸。
3. 为不同缓存类别设置最大条目数，超限后淘汰最早项，控制常驻内存。

这一步的价值不只是“少建几个 Surface”，而是把项目里分散的渲染复用需求收敛到一个稳定入口，降低后续维护成本。

### 3.2 场景级 UI 渲染去重复创建

以下场景和 UI 已切换到 `SurfaceCache`：

- `airwar/scenes/menu_scene.py`
- `airwar/scenes/login_scene.py`
- `airwar/scenes/pause_scene.py`
- `airwar/ui/reward_selector.py`

优化点包括：

1. 背景渐变改为按尺寸缓存，而不是每帧重新绘制。
2. 粒子发光改为按尺寸和量化透明度复用。
3. 面板、选项框、边框、发光框改为复用缓存 Surface。
4. 粒子更新由“遍历副本并原地删除”改为“构建活动列表”，减少列表修改成本和逻辑噪音。

### 3.3 背景渲染器优化

文件：

- `airwar/game/rendering/background_renderer.py`

本次处理：

1. 接入统一渐变缓存与粒子光晕缓存。
2. 清理缓存时统一调用 `SurfaceCache.clear()`，避免各模块各自维护局部缓存。
3. 粒子更新逻辑改为活动列表收集，避免在迭代过程中修改原列表。

### 3.4 精灵绘制缓存化

文件：

- `airwar/utils/sprites.py`
- `airwar/entities/bullet.py`

本次将以下绘制改为“首次构建，后续复用”：

- 玩家机
- 普通敌机
- Boss
- 单发子弹
- 扩散子弹
- 激光子弹

具体做法：

1. 保留原有绘制细节，但把手绘逻辑提取到 `_render_*` 内部函数。
2. 对外 `draw_*` 只负责根据尺寸和状态命中缓存后直接 `blit`。
3. 以尺寸、血量阶段、子弹类型作为缓存键，平衡表现变化与复用率。
4. 为精灵缓存增加最大条目限制，避免尺寸组合过多时长期占用内存。

这一步是本轮最关键的 CPU 优化之一，因为它直接命中了游戏主战斗循环里最频繁的绘制路径。

### 3.5 HUD 与常驻界面优化

文件：

- `airwar/game/systems/hud_renderer.py`
- `airwar/ui/buff_stats_panel.py`
- `airwar/entities/enemy.py`
- `airwar/game/mother_ship/progress_bar_ui.py`

本次处理：

1. Boss 血条字体改为初始化时创建，避免逐帧构造字体对象。
2. Buff 面板启用签名缓存，仅在 Buff 列表或汇总内容变化时重建整块 Surface。
3. Boss 的 `! WARNING !` 和 `ESCAPING...` 文本改为复用预渲染 Surface，仅调整透明度。
4. 母舰进度条图标改为初始化时构建一次并重复使用。
5. 进度条底板、边框、高亮继续复用统一矩形缓存。

### 3.6 入场动画与过渡效果优化

文件：

- `airwar/game/rendering/game_renderer.py`

本次处理：

1. 入场动画缓冲区改为可复用 Surface，避免每帧新建中间画布。
2. 淡入淡出遮罩改为统一缓存，而不是逐帧重新创建透明遮罩。

## 四、结果与收益

本次优化带来的直接收益主要集中在以下方面：

1. **减少 CPU 峰值**: 逐帧手绘和逐帧 Surface 创建显著减少，渲染主路径更稳定。
2. **降低内存抖动**: 统一缓存后，频繁的短生命周期 Surface 分配明显下降。
3. **避免缓存反噬**: 通过透明度量化和缓存上限，控制长期运行时的内存增长风险。
4. **改善结构可维护性**: 场景、HUD、背景、精灵不再各自复制缓存逻辑，后续优化可沿统一入口继续扩展。

验证结果：

- 当前 warning 来自 `pygame/pkgdata.py` 对 `pkg_resources` 的弃用提示，属于第三方依赖提示，不是本次优化引入的问题。

## 六、剩余可继续观测的点

虽然本次已经覆盖了主要热点，但仍建议后续继续观察以下区域：

1. `draw_ripple()` 一类动态图形仍然是即时绘制逻辑，适合在确认真实瓶颈后再做二次缓存。
2. `GameRenderer` 的缩放过程仍依赖 `pygame.transform.smoothscale()`，若后续入场动画成为新热点，可继续做分辨率或帧级量化。
3. 若未来引入更多尺寸变化、皮肤或主题，需同步审视缓存键设计，避免复用率下降。

---

## 七、结论

本次优化不是简单地“把几个 Surface 搬进缓存”，而是建立了一个覆盖场景层、HUD 层、背景层、精灵层的统一渲染复用策略，并补上了缓存有界治理。  
在不改变核心玩法和现有测试行为的前提下，项目运行时资源消耗已经得到系统性收敛，后续若继续做性能专项，可以在现有结构上逐步细化，而不需要重新推翻渲染实现。
