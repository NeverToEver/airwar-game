## 一、项目当前状况分析

### 1.1 项目概述

Air War 是一款基于 Python/Pygame 的飞行射击游戏，采用模块化架构设计，包含以下核心模块：

| 模块 | 路径 | 功能 |
|------|------|------|
| 窗口管理 | `airwar/window/` | 窗口创建与事件处理 |
| 渲染系统 | `airwar/game/rendering/` | 游戏画面渲染 |
| 实体系统 | `airwar/entities/` | 玩家/敌人/子弹实体 |
| 游戏逻辑 | `airwar/scenes/` | 游戏场景与状态管理 |
| 输入处理 | `airwar/input/` | 用户输入抽象 |
| 增益系统 | `airwar/game/buffs/` | 技能与增益效果 |
| 数据持久化 | `airwar/utils/` | 用户数据与配置 |

### 1.2 已完成工作

- ✅ BOSS 全向攻击功能已实现（上下左右四方向随机攻击）
- ✅ BOSS 生命值与存活时间已调整（生命值2000，逃跑时间60秒）
- ✅ 约 150 个单元测试覆盖核心功能

## 二、具体优化目标与优先级

### 2.1 优化目标矩阵

| 优先级 | 优化项 | 预期收益 | 工作量 |
|--------|--------|----------|--------|
| **P0-高** | 提取攻击方向配置为共享方法 | 代码可维护性提升 | 30分钟 |
| **P0-高** | 新增攻击方向测试用例 | 功能验证覆盖率 | 45分钟 |
| **P1-中** | 整理魔法数字至配置文件 | 代码可读性提升 | 20分钟 |
| **P1-中** | 添加输入验证与异常处理 | 健壮性提升 | 60分钟 |
| **P2-低** | 优化 Surface 缓存 | 渲染性能提升 | 30分钟 |

### 2.2 详细优化目标

#### 目标1: 消除重复代码（方向配置）

**当前问题**：
```python
# _spread_attack 中
direction_offsets = {'down': ..., 'left': ..., 'right': ..., 'up': ...}

# _aim_attack 中
direction_sources = {'down': ..., 'left': ..., 'right': ..., 'up': ...}

# _wave_attack 中
direction_sources = {'down': ..., 'left': ..., 'right': ..., ...}
```

**优化目标**：提取为类方法或类属性，消除3处重复

#### 目标2: 新增测试用例

**当前覆盖**：7个 BOSS 测试用例，无攻击方向测试

**目标覆盖**：
- attack_direction 初始化测试
- 随机方向选择分布测试
- 各方向攻击发射位置测试
- 各方向子弹飞行方向测试

#### 目标3: 魔法数字整理

**已识别魔法数字**：
- `fire_cooldown = 8` → `PLAYER_FIRE_RATE`
- `damage = 20/30` → `PLAYER_COLLISION_DAMAGE`
- `speed = 5.0/7.0` → 提取为 BOSS_BULLET_SPEED 常量
- `30, 45, 22.5` → 提取为攻击角度常量
- `500` → 提取为攻击距离常量

#### 目标4: 健壮性增强

**检查范围**：
- 输入参数验证（负数、None、边界值）
- 异常处理（文件加载失败、资源耗尽）
- 错误消息清晰度

## 六、风险评估与应对

### 6.1 风险识别

| 风险项 | 概率 | 影响 | 风险等级 |
|--------|------|------|----------|
| 重构引入新 Bug | 中 | 高 | 🟠 中 |
| 测试覆盖不完整 | 高 | 中 | 🟠 中 |
| 性能优化未达预期 | 低 | 中 | 🟢 低 |
| 依赖外部工具兼容性 | 低 | 低 | 🟢 低 |

### 6.2 应对策略

| 风险 | 应对措施 |
|------|----------|
| 重构引入新 Bug | 1. 先添加测试用例覆盖现有功能<br>2. 小步重构，每次验证<br>3. 保留原方法作为备选 |
| 测试覆盖不完整 | 1. 优先添加新增功能的测试<br>2. 审查现有测试的有效性<br>3. 补充边界情况测试 |
| 性能优化未达预期 | 1. 先进行性能基准测试<br>2. 量化优化前后差异<br>3. 必要时回退优化 |

### 6.3 回滚方案

如优化过程中出现问题：
1. **代码回滚**：使用 Git 回退到修改前版本
2. **测试回滚**：保留修改前通过的测试子集
3. **功能验证**：对比修改前后的游戏行为
